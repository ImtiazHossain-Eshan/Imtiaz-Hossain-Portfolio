## The number that looks great

Twenty-eight statistical features (lexical, syntactic, readability, distributional),
a Random Forest, and 97.40% accuracy on the HC3 benchmark distinguishing human
from ChatGPT text. On paper, solved.

Then I changed the generator, and the whole thing fell apart.

## The features

The premise is that machine writing has measurable statistical fingerprints:
narrower vocabulary, more uniform sentence lengths, more predictable formatting.
The top discriminative features by mutual information were revealing:

| Feature | MI score |
| --- | --- |
| whitespace ratio | 0.255 |
| automated readability | 0.227 |
| vocabulary richness | 0.227 |
| Zipf's coefficient | 0.212 |

That whitespace ratio tops the list is telling: LLMs have deterministic
formatting habits, and those habits leak.

## The cliff

Here is the result the headline number hides. Train on HC3 (ChatGPT), then
evaluate on M4 text generated by **Bloomz**:

| Test set | Accuracy |
| --- | --- |
| HC3 (in-domain, ChatGPT) | 97.40% |
| M4 Dev (Bloomz) | 20-30% |

Below chance. On a binary-ish problem, the model does *worse* than a coin flip,
because it confidently applies ChatGPT's fingerprint to text that does not have
it, and systematically labels Bloomz output as human.

## Why generalization is the wrong expectation

Each language model has its own training data, architecture, and decoding
strategy, so each produces a distinct statistical profile. A detector trained on
one generator learns *that generator's* signature, not a universal "machine-ness."
There is no shared boundary to find, because there is no shared distribution.

This is not a bug in the features. It is a property of the problem. The features
work exactly as designed; they just describe ChatGPT specifically.

## What honest reporting looks like

The weak version of this project reports 97.40% and stops. The useful version
reports 97.40% **and** the 20-30% cliff, because the second number is what tells
a practitioner whether they can deploy this. A detector that only works on the
generator it was trained on is a detector with a very short shelf life, since the
models keep changing.

The interpretable upside: unlike a GPU-hungry neural detector, this pipeline runs
on CPU and tells you *which* features drove each decision. You can explore the
feature importances and the generalization gap in the
[playground](/playground/text-detection).
